# Development Plan: "The Physical AI Handbook"

This document outlines the architectural and development plan for building "The Physical AI Handbook" using Docusaurus. It translates the project's constitution and specification into an actionable roadmap for setup, content creation, and maintenance.

---

## 1. Docusaurus Setup and Configuration

This section details the step-by-step process for initializing and configuring the Docusaurus project.

### 1.1. Prerequisites
- **Node.js:** Version 18.0 or higher.
- **Yarn or npm:** A modern version of either package manager.

### 1.2. Installation
The Docusaurus site will be scaffolded inside a dedicated directory named `handbook-site` to keep it separate from other project artifacts like specs and plans.

1.  **Initialize the Docusaurus Project:**
    Open a terminal in the project root and run the following command:
    ```bash
    npx create-docusaurus@latest handbook-site classic
    ```
2.  **Navigate into the new directory:**
    ```bash
    cd handbook-site
    ```
3.  **Start the development server** to confirm successful installation:
    ```bash
    npm run start
    ```
    The site should now be accessible at `http://localhost:3000`.

### 1.4. Sidebar Configuration (`sidebars.js`)
To support the auto-generated, file-based chapter and lesson structure, the `sidebars.js` file must be configured as follows:

```javascript
// handbook-site/sidebars.js

/** @type {import('@docusaurus/plugin-content-docs').SidebarsConfig} */
const sidebars = {
  // By default, Docusaurus generates a sidebar from the docs folder structure
  tutorialSidebar: [{type: 'autogenerated', dirName: '.'}],
};

export default sidebars;
```

## 2. Content Development Phases

This phased approach ensures each lesson is high-quality, accurate, and consistent.

*   **Phase 1: Outline and Plan**
    *   For each lesson, create a detailed outline in a separate document.
    *   Define learning objectives, required components, and a sketch of the final project's functionality. This ensures clarity before any prose is written.

*   **Phase 2: Draft Content**
    *   Write the first draft of the lesson's text and code in a Markdown file (`.mdx`).
    *   Strictly follow the `Lesson Format` defined in the specification. Focus on clear, step-by-step instructions.

*   **Phase 3: Create Assets**
    *   Produce all required visual aids. This is a critical step.
    *   **Wiring Diagrams:** Create clean, readable diagrams using Fritzing or a similar tool.
    *   **Photos:** Take high-resolution photographs of the actual completed circuit.
    *   **GIFs/Videos:** Record short animated GIFs or videos showing the project in action. This is the "proof" that the project works.
    *   Store all assets according to the file structure defined in Section 3.

*   **Phase 4: Technical Review and Validation**
    *   **Physical Build:** A technical reviewer must build the project from scratch using *only* the drafted lesson.
    *   **Code Validation:** All code must be copied from the draft and run on the target hardware to ensure it works without modification.
    *   **Feedback:** The reviewer provides feedback on errors, ambiguities, or potential improvements.

*   **Phase 5: Editorial Review**
    *   The draft is proofread for spelling, grammar, and style.
    *   A reviewer checks that the tone aligns with the "Brand Voice" defined in the constitution (encouraging, patient, clear).

*   **Phase 6: Integration and Finalization**
    *   Incorporate feedback from both technical and editorial reviews.
    *   Place the final `.mdx` file into the correct location within the `handbook-site/docs/` directory.
    *   Embed all finalized assets using relative paths.
    *   Add Docusaurus-specific formatting like admonitions (`::note`, `::danger`, etc.).
    *   Submit a pull request for the completed lesson.

## 3. File Structure

```

├── docs/

│       └── lesson-1-3-automatic-night-light.mdx
```
